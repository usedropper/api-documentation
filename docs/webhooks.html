Webhooks

tip

Webhooks allow you to set up a notification system that can be used to receive updates on certain requests made to the Dropper API. We recommend that you use webhooks to provide value to your customers over using callbacks or polling.


Webhooks vs Pollingâ€‹

Image courtesy: Fikayo Adepoju

Polling is when your application periodically calls an API to check if an event has occurred or new data exists. Webhooks, on the other, send data down to your application when an event occurs in real-time.

To capture the difference between these two approaches with a relatable example, polling is like going to the post office to check if you have new mail. Using webhooks is like having mail delivered to your house every time you have new mail simply by giving the postman your house address. Polling is more resource-heavy compared to webhooks, as it can take multiple network requests before new information is discovered, while webhooks only make network requests when there is new information.

With webhooks, a resource server, Dropper in this case, sends updates to your server when the status of your request changes. A change in status of a request is known as an event. You'll typically listen to these events on a POST endpoint called your webhook URL. Read more...


Create a webhook URLâ€‹

A webhook URL is simply a POST endpoint that a resource server sends updates to. After creating your webhook, you can set your webhook URL on the Dropper dashboard.

When your webhook URL receives an event, it needs to parse and acknowledge the event. Acknowledging an event means returning a 200 OK in the HTTP header. Without a 200 OK in the response header, we'll send 2 retries 3 minutes apart.

Create a webhook
// Using express
app.post("/your/webhook/url", function(req, res) {
    const event = req.body;
    // Process event  
    
    res.send(200);
});
caution

If you have a long-running task in your webhook, you should acknowledge the event before executing the task to prevent a request timeout and an automatic error response from your server. Without a timely 200 OK response, we retry the event as described above.


Verify event originâ€‹

Since your webhook URL is publicly available, you need to verify that events originate from Dropper and not a bad actor. You can do this by signature validation.

Webhook signature validation
const crypto = require('crypto');
const secret = process.env.DROPPER_API_KEY;

// Using express
app.post("/your/webhook/url", function(req, res) {
    // Validate event
    const hash = crypto.createHmac('sha512', secret).update(JSON.stringify(req.body)).digest('hex');
    if (hash == req.headers['x-dropper-signature']) {
        const event = req.body;
        // Process event  
    }
    res.send(200);
});

Webhook checklistâ€‹

Now that we've covered all there is to know about using webhooks with Dropper's API, here is a recap to ensure you have a great experience:

  • Add the webhook URL on your Dropper dashboard.
  • Ensure your webhook URL is publicly available (localhost URLs cannot receive events).
  • Test your webhook to ensure you're getting the JSON body and returning a 200 OK HTTP response.
  • If your webhook function has long-running tasks, you should first acknowledge receiving the webhook by returning a 200 OK before proceeding with the tasks.
  • If we don't get a 200 OK HTTP response from your webhooks, we flagged it as a failed attempt and retry twice 3 minutes apart.
